# 2026-04-16 17:06 — Release 1.12.1: formato 24hs forzado en inputs datetime

## Tema

Bug reportado por Charly: en el form de PDS (y otros módulos con `<input type="datetime-local">`), el campo "Fecha y hora de inicio" y "Finalizado" aparecían en formato 12hs con AM/PM en algunos browsers/SO (screenshot mostraba `04/16/2026 04:46:14 PM`), cuando la UI del cálculo horario de abajo (duración bruta/neta) asume 24hs. Esto rompía la coherencia visual y aumentaba el riesgo de que el usuario cargue mal una hora.

## Diagnóstico

El input HTML `<input type="datetime-local">` decide el formato 12/24hs **según el locale del browser/SO del usuario**. No hay atributo nativo (`lang`, `locale`, `pattern`) que lo fuerce — Chromium y WebKit ignoran explícitamente `lang` para este tipo de input. En Windows con el idioma del sistema en inglés-US, siempre aparece AM/PM.

Inputs afectados en la app (4 módulos):
- `app/modules/CrmPedidosServicio/views/form.php:240,246` — fecha_inicio y fecha_finalizado.
- `app/modules/CrmPresupuestos/views/form.php:379` — fecha.
- `app/modules/CrmAgenda/views/form.php:71,76` — inicio y fin.
- `app/modules/RxnLive/views/dataset.php:679-683` — inputs de filtros avanzados (renderizados dinámicamente via `innerHTML`).

## Solución elegida

Incorporar **Flatpickr 4.6.13** (jsdelivr, ~20KB sin deps) como picker estandarizado con config fija 24hs + segundos + locale es. Wrapper global `public/js/rxn-datetime.js` que auto-inicializa sobre todos los `input[type="datetime-local"]` e `input[type="date"]` al `DOMContentLoaded`.

Alternativas descartadas:
- **Input text + máscara custom**: se pierde el picker nativo, más código, peor UX.
- **Forzar `lang="es-AR"`**: ignorado por Chromium y WebKit para datetime-local. No funciona.
- **Tempus Dominus / Air Datepicker / jQuery UI**: suman jQuery y/o más peso sin ventaja funcional.

## Qué se hizo

### 1. Wrapper global (`public/js/rxn-datetime.js`)

- Config fija: `enableTime: true, time_24hr: true, enableSeconds: true, dateFormat: "Y-m-d H:i:S", allowInput: true, locale: es`.
- Auto-init en `DOMContentLoaded`.
- API pública `window.RxnDateTime`:
  - `init(el)` — inicializa un input.
  - `initAll(container)` — inicializa todos los inputs dentro del container (para render dinámico).
  - `setValue(input, value)` — setea valor programático respetando la instancia Flatpickr (un `.value = "..."` directo no actualiza el picker).
- Degrada gracefully: si Flatpickr no cargó (offline, CDN bloqueado), los inputs nativos siguen funcionales.
- El `onChange` re-dispacha evento `input` para mantener compat con listeners legacy que escuchan ese evento (ej: calculadora horaria del PDS).

### 2. Admin layout

`app/shared/views/admin_layout.php`:
- CSS de Flatpickr en `<head>`.
- JS core + locale es + wrapper `rxn-datetime.js` antes de los scripts de módulo. Orden: Bootstrap → Flatpickr → locale → rxn-datetime → módulo.

### 3. Compat con el botón "Ahora" del PDS

`public/js/crm-pedidos-servicio-form.js` línea 310-320: el listener del botón `btn-finalizado-ahora` antes hacía `endInput.value = localIso`. Con Flatpickr activo eso no se refleja en el picker. Ahora usa `RxnDateTime.setValue()` con formato "YYYY-MM-DD HH:MM:SS" (espacio) y con fallback a `.value` directo por si el wrapper no está presente.

### 4. Re-init en RxnLive

`app/modules/RxnLive/views/dataset.php` línea ~739: después de `div.innerHTML = html` (que renderiza los filtros avanzados de columna, incluyendo inputs datetime-local dinámicos), se invoca `window.RxnDateTime.initAll(div)` para que esos inputs recién inyectados también queden con Flatpickr.

### 5. Convención registrada en CLAUDE.md

Nueva sección **"Regla UI: inputs de fecha/hora SIEMPRE en formato 24hs"** (antes de "Vocabulario acordado con Charly"):
- Explica el por qué (locale del SO).
- Documenta la API del wrapper.
- Indica cuándo NO usar `datetime-local` / `time` crudos.
- Regla dura: cualquier AM/PM visible en la UI es bug.

## Por qué

Charly reportó que el PDS mostraba AM/PM en el input de fecha inicio, y que eso era inconsistente con la sección de cálculo horario de abajo (que siempre está en 24hs + segundos). Cuando le propuse el plan, pidió además:
1. Mantener los segundos (el esquema actual los necesita para la duración neta).
2. Aplicarlo a **todas** las vistas donde aparezca (no sólo PDS).
3. Registrar la convención en CLAUDE.md como regla transversal para próximas sesiones.

## Impacto

- **Visual**: todos los inputs de fecha/hora ahora muestran el picker de Flatpickr con formato 24hs y segundos, consistente en Chrome, Firefox, Edge, Safari y cualquier SO/locale.
- **Funcional (PDS)**: la calculadora horaria sigue funcionando idéntica — el listener de `input` se dispara igual gracias al re-dispatch del wrapper. Botón "Ahora" funciona con el API de Flatpickr.
- **Funcional (Presupuestos, Agenda)**: no requirieron cambios — el auto-init los cubre.
- **Funcional (RxnLive filtros)**: los filtros de rango de fecha en DB ahora tienen el mismo picker.
- **Backend**: sin cambios. El parser de `PedidoServicioController::parseDateTimeInput` ya aceptaba ambos formatos (`Y-m-d H:i:s` y `Y-m-d\TH:i:s`).

## Decisiones tomadas

- **Flatpickr vs custom**: Flatpickr por peso/UX/mantenimiento.
- **Versión lock 4.6.13**: evita sorpresas de breaking changes en minors.
- **CDN jsdelivr**: mismo provider que Bootstrap (ya cargado), mismo CDN warm en cache para la mayoría de los usuarios.
- **Wrapper global en vez de init por módulo**: evita repetir config en cada vista; convención centralizada y auditable.
- **Re-dispatch de `input`** en `onChange`: mantiene compat con el JS legacy del PDS sin obligar a refactorizarlo.
- **`allowInput: true`**: permite que el usuario siga pudiendo tipear si prefiere — no obliga a usar el picker.
